### <a id="introduction">Введение</a>
В современных спортивных школах управление данными играет ключевую роль в обеспечении эффективности и конкурентоспособности. Успешная деятельность требует систематизации информации о спортсменах, тренерах, администраторах, соревнованиях и инвентаре. Ручное ведение учёта уже не способно справляться с растущими объёмами данных, что приводит к ошибкам, потерям времени и снижению качества обслуживания клиентов. Поэтому создание автоматизированной базы данных становится важной задачей для спортивной школы, так как оно обеспечивает быструю обработку данных и повышает точность учёта.
В данной работе рассматривается процесс разработки базы данных для спортивной школы, целью которой является создание надёжного и функционального инструмента для хранения и управления данными о спортсменах, тренерах, инвентаре и соревнованиях. Такой подход позволит улучшить контроль за операциями, сократить временные затраты на обработку информации и повысить общую продуктивность работы организации.

#### <a id="target">Цель работы</a>
Целью данной курсовой работы является разработка базы данных для торговой организации, которая обеспечит автоматизацию процессов учёта и анализа данных. База данных должна стать надёжным инструментом для хранения и управления информацией о товарах, клиентах, поставщиках и продажах. Она должна позволять:
- Вести учёт товаров с указанием их категории, цены, количества на складе и других характеристик.
- Регистрировать продажи и анализировать их для выявления трендов и прогнозирования спроса.
- Хранить данные о клиентах, включая контактную информацию и историю покупок.
- Формировать аналитические отчёты для принятия стратегических решений.

Кроме того, база данных должна быть масштабируемой, чтобы справляться с ростом объёмов данных, и защищённой, чтобы предотвращать несанкционированный доступ к информации. Эти требования делают разработку базы данных сложной, но крайне важной задачей для современной торговой организации.

   **Описание предметной области**
Современные торговые организации сталкиваются с растущими объемами данных, которые необходимо эффективно обрабатывать и анализировать для поддержания конкурентоспособности. Управление такими данными вручную становится неэффективным и чревато ошибками, что может привести к снижению качества обслуживания клиентов и упущенным возможностям для бизнеса. Автоматизация процессов учёта и управления данными становится не просто желательной, а насущной необходимостью.

Торговая организация работает в условиях высокой конкуренции, где успех зависит от способности быстро реагировать на изменения спроса, оптимизировать запасы товаров и улучшать качество взаимодействия с клиентами. База данных позволяет систематизировать информацию о товарах, клиентах, поставщиках и продажах, предоставляя удобный инструмент для анализа и принятия решений. 

Например, благодаря автоматизации можно:
- Отслеживать наличие товаров на складе в режиме реального времени.
- Анализировать данные о продажах для выявления наиболее популярных товаров.
- Формировать персонализированные предложения для клиентов на основе их истории покупок.
- Оптимизировать взаимодействие с поставщиками для своевременного пополнения запасов.

Автоматизированная база данных также позволяет интегрироваться с другими системами, такими как CRM (Customer Relationship Management), системы бухгалтерского учёта и интернет-магазины. Это создает единую экосистему, которая обеспечивает бесперебойную работу всех бизнес-процессов.
      
    - **Функции решаемых задач:**
1. **Учёт товаров:**
   - Ведение подробных данных о каждом товаре, включая название, категорию, цену, количество на складе и описание. 
   - Возможность добавления фотографий или других медиафайлов для каждого товара.
   - Интеграция с системами контроля запасов для автоматического пополнения товаров при достижении минимального уровня.
   - Хранение информации о поставщиках каждого товара, включая их контактные данные и условия сотрудничества.

2. **Регистрация продаж:**
   - Запись данных о каждой продаже, включая дату, время, список товаров, их количество и итоговую сумму.
   - Возможность добавления фотографий или видеоматериалов, связанных с продажей (например, чеков).
   - Интеграция с системами лояльности для начисления бонусов клиентам.
   - Анализ данных о продажах для выявления сезонных трендов и прогнозирования спроса.

3. **Учёт клиентов:**
   - Сохранение данных о клиентах, включая ФИО, контактную информацию, адрес проживания и историю покупок.
   - Возможность добавления отзывов и оценок клиентов для улучшения качества обслуживания.
   - Интеграция с CRM-системами для формирования персонализированных предложений и маркетинговых кампаний.

4. **Учёт поставщиков:**
   - Хранение данных о поставщиках, включая название компании, контактную информацию, ассортимент товаров и условия сотрудничества.
   - Возможность отслеживания сроков поставок и качества товаров.
   - Интеграция с системами закупок для автоматического оформления заказов.

2. **Перечень входных данных.** Для эффективной работы системы необходимы следующие входные данные:
1. **Товары:**

2. **Клиенты:**

3. **Поставщики:**

   - Ассортимент поставляемых товаров.

Для решения этой проблемы необходимо использовать механизмы оптимизации, такие как индексация, партиционирование таблиц и кэширование часто запрашиваемых данных.

    2. **Ограничение по доступу:**
   - Данные, такие как личная информация о клиентах или данные о продажах, должны быть защищены от несанкционированного доступа. Это ограничение требует реализации надёжных механизмов авторизации и разграничения прав пользователей. Например, только администраторы должны иметь доступ к данным о финансах, а менеджеры — к данным о клиентах.

   - Чтобы избежать потери данных, необходимо разработать систему резервного копирования. Это ограничение предполагает наличие регулярных бэкапов для предотвращения потерь информации в случае технических сбоев. Рекомендуется использовать комбинацию логического и физического резервного копирования.

    4. **Ограничение по производительности:**
   - При работе с большими объёмами данных важно учитывать производительность системы. Например, использование индексов может значительно ускорить выполнение запросов, но также увеличивает нагрузку на сервер при вставке новых данных. Поэтому необходимо тщательно планировать использование индексов.

    5. **Взаимодействие с другими программами.** База данных спортивной школы может взаимодействовать с различными программами и модулями для расширения функциональности и повышения удобства работы:

#### <a id="infological_model">Инфологическая (концептуальная) модель базы данных</a>
1. **Выделение информационных объектов.**
    - **Спортсмен** - информация о спортсменах, обучающихся в школе.
    - **Тренер** - данные о тренерах, работающих в школе.
    - **Соревнование** - данные о соревнованиях, в которых участвуют спортсмены.
    - **Инвентарь** - информация об инвентаре, используемом в школе.

#### Определение отношений
1. **Тренер - Спортсмен:**
   - **Связь:** один ко многим.
   - **Описание:** Один тренер может обучать множество спортсменов, но каждый спортсмен привязан только к одному тренеру. Это позволяет четко отслеживать, кто за кого отвечает.

2. **Спортсмен - Соревнование:**
   - **Связь:** один ко многим.
   - **Описание:** Один спортсмен может участвовать в множестве соревнований, но каждое соревнование связано с конкретным списком участников.

3. **Инвентарь - Спортсмен:**
   - **Связь:** один ко многим.
   - **Описание:** Один спортсмен может использовать множество единиц инвентаря, но каждая запись об использовании связана с конкретным спортсменом.

3. **Определение отношений и мощности отношений между объектами.**
    - **Тренер - Спортсмен:**
        - **Связь:** один ко многим.
        - **Описание:** Один тренер может обучать множество спортсменов.
    - **Спортсмен - Соревнование:**
        - **Связь:** один ко многим.
        - **Описание:** Один спортсмен может участвовать в множестве соревнований.
    - **Инвентарь - Спортсмен:**
        - **Связь:** один ко многим.
        - **Описание:** Один спортсмен может использовать множество единиц инвентаря.

4. **Построение концептуальной модели.**
    - **Тренер** (ID_тренера, ФИО, специализация, контактная информация), связан с **Спортсмен** (ID_спортсмена, ФИО, дата рождения, контактная информация).
    - **Спортсмен** связан с **Соревнование** (ID_соревнования, название, дата проведения, место проведения).
    - **Инвентарь** (ID_инвентаря, название, описание, количество), связан с **Спортсмен** через отношение "один ко многим".

#### <a id="logical_structure">Логическая структура базы данных</a>
**Определение таблиц и их атрибутов**
1. **Таблица "Тренеры"**
    - **ID_тренера** (Primary Key) - уникальный идентификатор тренера (целое число).
    - **ФИО** - фамилия, имя и отчество тренера (строка).
    - **Специализация** - специализация тренера (строка).
    - **Контактная информация** - контактные данные тренера (строка).

2. **Таблица "Спортсмены"**
    - **ID_спортсмена** (Primary Key) - уникальный идентификатор спортсмена (целое число).
    - **ФИО** - фамилия, имя и отчество спортсмена (строка).
    - **Дата рождения** - дата рождения спортсмена (дата).
    - **Контактная информация** - контактные данные спортсмена (строка).
    - **ID_тренера** (Foreign Key) - ссылка на тренера спортсмена (целое число).

3. **Таблица "Соревнования"**
    - **ID_соревнования** (Primary Key) - уникальный идентификатор соревнования (целое число).
    - **Название** - название соревнования (строка).
    - **Дата проведения** - дата проведения соревнования (дата).
    - **Место проведения** - место проведения соревнования (строка).

4. **Таблица "Инвентарь"**
    - **ID_инвентаря** (Primary Key) - уникальный идентификатор инвентаря (целое число).
    - **Название** - название инвентаря (строка).
    - **Описание** - описание инвентаря (строка).
    - **Количество** - количество единиц инвентаря (целое число).

#### <a id="physical_structure">Физическая структура базы данных</a>

1. **Выбор типов данных**
    - **INT** — используется для целочисленных значений, таких как идентификаторы записей.
    - **VARCHAR(255)** — применяется для хранения текстовых данных переменной длины.
    - **DATE** — используется для хранения дат.

2. **Оптимизация индексов**
Индексы являются ключевым элементом в оптимизации базы данных, так как они позволяют ускорить доступ к данным и облегчают выполнение запросов, таких как поиск, фильтрация и сортировка. Индексы создаются для полей, которые часто участвуют в поисковых запросах и соединениях таблиц.

1. **Первичные ключи:**
   - Автоматически индексируются и обеспечивают быстрый поиск записей.

2. **Внешние ключи:**
   - Например, индекс для поля `ID_категории` в таблице `Товары`:
     CREATE INDEX idx_товары_категория ON Товары (ID_категории);
3. **Дополнительные индексы:**
   - Для поля `Дата_продажи` в таблице `Продажи`:
CREATE INDEX idx_продажи_дата ON Продажи (Дата_продажи);

#### Механизмы хранения
PostgreSQL использует механизм хранения данных, поддерживающий многоверсионность (MVCC), что позволяет эффективно управлять транзакциями и конкурентным доступом к данным. MVCC гарантирует согласованность данных и поддерживает высокий уровень надежности.

#### Параметры хранения
Для повышения производительности можно настроить следующие параметры PostgreSQL:
- **shared_buffers:** Размер памяти, выделяемой для кэширования данных.
- **work_mem:** Размер памяти для операций сортировки и соединений.
- **maintenance_work_mem:** Размер памяти для операций обслуживания, таких как создание индексов.

-- Создание функции для предотвращения самокоучинга
CREATE FUNCTION public.prevent_self_coaching() RETURNS TRIGGER AS $$
BEGIN
    IF EXISTS (
        SELECT 1 
        FROM coaches c
        JOIN students s ON c.user_id = s.user_id
        WHERE c.id = NEW.coach_id AND s.id = NEW.student_id
    ) THEN
        RAISE EXCEPTION 'Тренер не может быть студентом для самого себя';
    END IF;
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

-- Создание триггера для предотвращения самокоучинга
CREATE TRIGGER check_self_coaching
BEFORE INSERT OR UPDATE ON public.schedule
FOR EACH ROW
EXECUTE FUNCTION public.prevent_self_coaching();


#### Разработка запросов
Запросы для управления базой данных и получения информации помогут обеспечить работу торговой организации. Примеры запросов включают:

1. **Вывод списка всех товаров по категориям с указанием цены и наличия:**
SELECT t.Название AS Товар, k.Название_категории AS Категория, t.Цена, t.Количество_на_складе
FROM Товары t
JOIN Категории k ON t.ID_категории = k.ID_категории;

2. **Запрос для получения информации о клиентах и их последних покупках:**
SELECT c.ФИО, c.Телефон, p.Дата_продажи, p.Общая_сумма
FROM Клиенты c
JOIN Продажи p ON c.ID_клиента = p.ID_клиента
ORDER BY p.Дата_продажи DESC;

3. **Запрос для получения общего дохода за период:**
SELECT SUM(Общая_сумма) AS Общий_доход
FROM Продажи
WHERE Дата_продажи BETWEEN '2024-01-01' AND '2024-12-31';

4. **Запрос для анализа популярности товаров:**
SELECT t.Название, SUM(sp.Количество) AS Всего_продано
FROM Состав_продажи sp
JOIN Товары t ON sp.ID_товара = t.ID_товара
GROUP BY t.Название
ORDER BY Всего_продано DESC;

#### Назначение прав доступа
Назначение прав доступа в PostgreSQL выполняется с помощью ролей и команд **GRANT** и **REVOKE**. Для торговой организации можно выделить следующие роли:
- **Администратор:** Полные права на создание, изменение и удаление данных.
- **Менеджер:** Может добавлять и обновлять данные о продажах и клиентах, но не имеет прав на удаление записей.
- **Сотрудник:** Может только просматривать информацию о товарах и клиентах.

Пример назначения прав доступа:
-- Создание ролей
CREATE ROLE Администратор WITH LOGIN PASSWORD 'password';
CREATE ROLE Менеджер WITH LOGIN PASSWORD 'password';
CREATE ROLE Сотрудник WITH LOGIN PASSWORD 'password';

-- Назначение прав для каждой роли
GRANT ALL PRIVILEGES ON ALL TABLES IN SCHEMA public TO Администратор;
GRANT SELECT, INSERT, UPDATE ON Товары, Клиенты, Продажи, Состав_продажи TO Менеджер;
GRANT SELECT ON Товары, Клиенты TO Сотрудник;
